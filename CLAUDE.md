## Commands

```bash
npm run dev      # start dev server (localhost:3000)
npm run build    # production build
npm run lint     # eslint
```

There are no tests. There is no `src/` directory — all source is at the repo root.

## Next.js 16 critical differences

**Read `node_modules/next/dist/docs/` before touching routing or middleware.** Key breaking changes from your training data:

- **Middleware is renamed `proxy`** — the file is `proxy.ts` at the root (not `middleware.ts`). Export a named `proxy` function, not `default`.
- **`params` and `searchParams` are Promises** — always `await params` before accessing route segments.
- **Global `PageProps<'/route'>` / `LayoutProps<'/route'>` helpers** — no import needed, generated by `next dev`/`next build`.

## Architecture

### Route groups

```
app/
  layout.tsx              ← root layout: Roboto font, NextIntlClientProvider, <html lang>
  (public)/               ← public site (Header + Footer from layout.tsx inside group)
    page.tsx              ← home page
    news/  programs/  info-sessions/  aspirasi/  about/
  admin/
    login/                ← unguarded login page
    (panel)/              ← auth-guarded panel (AdminSidebar, demo banner)
      page.tsx            ← dashboard
      aspirasi/  news/  programs/  info-sessions/  users/
```

`app/admin/(panel)/layout.tsx` is the **second auth check** (profile active + role). `proxy.ts` is the optimistic edge check; the layout is the hard guard. Both must be kept in sync.

### Data layer

- **`lib/data.ts`** — public read-only queries (published news, programs, sessions, stats). Falls back to `lib/fixtures.ts` when `isSupabaseConfigured` is false.
- **`lib/admin-data.ts`** — staff-only queries (all news, all aspirasi with filters, profiles, dashboard counts). Same fixture fallback pattern.
- **`isSupabaseConfigured`** (`lib/supabase/env.ts`) — checked at runtime. When `false`, writes are no-ops and reads return fixtures. This is demo mode, not an error state.
- **`lib/supabase/admin.ts`** — service-role client. Server-only. Used exclusively by `/admin/users` to create/disable auth accounts. Never import in client components.

### i18n

Cookie-based locale (no URL prefix). Default locale is `id` (Indonesian).
- Toggle: `i18n/locale.ts` → `setUserLocale()` server action → `router.refresh()`
- All UI strings live in `messages/id.json` and `messages/en.json`
- Per-post bilingual: news/programs/sessions have optional `*_en` columns. `lib/i18n-content.ts` → `pickLocalized(locale, primary, translated)` applies the EN→ID fallback.
- Server components: `getTranslations('Namespace')` (async). Client components: `useTranslations('Namespace')`.

### Supabase

Three clients — use the right one:
| Client | Import from | Use in |
|--------|-------------|--------|
| Server | `lib/supabase/server.ts` | Server components, Server Actions, Route Handlers |
| Browser | `lib/supabase/client.ts` | Client components (`"use client"`) |
| Admin | `lib/supabase/admin.ts` | Server Actions that need service role (users page only) |

### Admin server actions pattern

All write actions (`"use server"`) follow this pattern:
1. `zod` parse the input — return `{ ok: false, error: "invalid" }` on failure
2. `if (!isSupabaseConfigured) return { ok: true }` — no-op in demo mode
3. `await requireUser()` / `requireStaff()` — return `{ ok: false, error: "unauthorized" }` if no session
4. Run the Supabase query
5. `revalidatePath(...)` for all affected public + admin routes

### Brand tokens (Tailwind v4)

Defined in `app/globals.css` under `@theme`. Use these class names:
- `text-navy` / `bg-navy` / `border-navy` (+ `-700`, `-100`, `-50` shades)
- `text-amber` / `bg-amber` / `border-amber` (+ `-600`)
- `text-pink` / `bg-pink`
- `text-ink` (body), `text-muted` (secondary), `bg-canvas` (page bg), `border-navy-100` (dividers)

Tailwind v4 is configured via PostCSS (`postcss.config.mjs`). There is **no `tailwind.config.js`** — all theme tokens live in `globals.css @theme`.

### Adding new admin content type

Follow the pattern in `app/admin/(panel)/news/`:
1. Actions file: `actions.ts` (`"use server"`, zod schema, `saveX` / `deleteX`)
2. Form component: `components/admin/XForm.tsx` (`"use client"`, calls action, `ImageUploader`)
3. Row actions: `components/admin/XRowActions.tsx` (`"use client"`, calls delete + status toggle)
4. Pages: `page.tsx` (list), `new/page.tsx` (chooser or blank form), `[id]/edit/page.tsx`
5. Add data fetchers to `lib/admin-data.ts` with fixture fallback
6. Add public data fetcher to `lib/data.ts` if needed
7. `revalidatePath` both the admin and public routes in actions

### Image uploads

`components/admin/ImageUploader.tsx` — drag-drop to Supabase Storage or paste URL fallback (used when Storage not configured). Alt text is required on every image. Storage buckets: `news-images`, `program-images`, `session-images` (all public read, authenticated write — defined in `lib/constants.ts` `STORAGE_BUCKETS`).
